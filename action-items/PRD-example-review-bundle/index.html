01

Executive summary

The proposed product change at a glance.

Current1 fixed template

Prose sections with limited review affordances.

TargetAdaptive bundle

Composable blocks selected from the feature's review needs.

Key contractTraceable decisions

Stable IDs connect requirements, risks, decisions, and tests.

Recommendation

Treat HTML as a generated product surface rather than a decorated document. Keep the skill responsible for discovery and judgment while the generator owns deterministic structure, assets, navigation, accessibility, and validation.

02

Problem and evidence

Why this outcome matters now.

Problem statement

The existing PRD output is a fixed document surface. It cannot adapt its presentation to UI, workflow, API, data, or architecture decisions without authors recreating layout structure.

Evidence

  • Reviewers must reconstruct relationships mentally while scanning a long page.
  • Important requirements, risks, decisions, and validation outcomes are difficult to trace.
  • Repeated bordered cards obscure hierarchy instead of improving review speed.

Scope boundaries

Explicit limits for implementation planning.

Review scope boundaries

In scope

  • Reference-aligned document shell and responsive navigation
  • Purpose-specific renderers for common PRD blocks
  • Mermaid and native visual review surfaces
  • Print-safe output

Out of scope

  • Persisted comments or approvals
  • Editable PRD content in the generated artifact
  • Migration of previously generated PRDs
09

Decision log

Settled choices that shape delivery.

DEC-01

Bundles may contain companion assets.

Rationale: Portability means a self-contained directory, not necessarily one HTML file.

Related: REQ-01

DEC-02

The rich interactive PRD is the visual contract.

Rationale: Generated artifacts should share its spacing, hierarchy, navigation, and bounded-surface rules.

Related: REQ-02

DEC-03

Review interactions are local and read-only.

Rationale: Reviewers may inspect generated views without changing or persisting PRD data.

Related: REQ-03

DEC-04

Traceability is generated from manifest IDs.

Rationale: Relationships remain deterministic and validator-friendly.

Related: REQ-04

11

Risks and mitigations

Known failure modes and responses.

RISK-01

A partial bundle appears complete.

Mitigation: Validate first and publish only after all files are written.

Related: REQ-01

RISK-02

Generic renderers flatten meaningful hierarchy.

Mitigation: Use block-specific semantic markup and visual regression fixtures.

Related: REQ-02

RISK-03

A generated review surface is mistaken for editable product UI.

Mitigation: Label generated surfaces as review aids and never persist state.

Related: REQ-03

RISK-04

Generated links imply false coverage.

Mitigation: Validate references and require semantically reviewed relationships.

Related: REQ-04
